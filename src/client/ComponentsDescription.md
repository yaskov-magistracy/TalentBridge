## Общие компоненты (Shared)

### NavbarComponent
**Селектор:** `app-navbar`

Навигационная панель приложения. Отображает логотип и адаптируется в зависимости от роли пользователя.

**Входные параметры (`@Input`):**
| Параметр | Тип | Описание |
|----------|-----|----------|
| `role` | `'candidate' \| 'employer'` | Роль текущего пользователя (опционально) |

**Особенности:**
- Логотип является ссылкой на дашборд соответствующей роли
- При указанной роли показывает бейдж с ролью и кнопку "ВЫХОД"
- Использует градиентный текст для логотипа

**Пример использования:**
```html
<!-- Для кандидата -->
<app-navbar [role]="'candidate'"></app-navbar>

<!-- Для работодателя -->
<app-navbar [role]="'employer'"></app-navbar>

<!-- Без авторизации (только логотип) -->
<app-navbar></app-navbar>
```

### TechChipComponent
**Селектор:** `app-tech-chip`

Компонент для отображения технологии в виде "чипа" (badge).

**Входные параметры (`@Input`):**
| Параметр | Тип | Описание |
|----------|-----|----------|
| `name` | `string` | Название технологии (обязательный) |

**Стили:**
- Черная рамка в стиле wireframe
- Заглавный текст с расширенным межбуквенным интервалом
- Маленький размер шрифта

**Пример использования:**
```html
<app-tech-chip [name]="'React'"></app-tech-chip>
<app-tech-chip *ngFor="let tech of technologies" [name]="tech"></app-tech-chip>
```

### StatusBadgeComponent
**Селектор:** `app-status-badge`

Цветной бейдж для отображения статуса проверки.

**Входные параметры (`@Input`):**
| Параметр | Тип | Описание |
|----------|-----|----------|
| `status` | `'pending' \| 'passed' \| 'failed' \| 'approved' \| 'rejected'` | Статус для отображения |

**Поддерживаемые статусы:**
| Статус | Текст | Цвет |
|--------|-------|------|
| `pending` | ОЖИДАНИЕ | Желтый (amber) |
| `passed` | ПРОЙДЕНО | Зеленый (emerald) |
| `approved` | ПРОЙДЕНО | Зеленый (emerald) |
| `failed` | НЕ ПРОЙДЕНО | Красный (red) |
| `rejected` | НЕ ПРОЙДЕНО | Красный (red) |

**Пример использования:**
```html
<app-status-badge [status]="'passed'"></app-status-badge>
<app-status-badge [status]="submission.status.autoTests"></app-status-badge>
```

### ReviewProgressComponent
**Селектор:** `app-review-progress`

Визуализация трехэтапного процесса проверки решения.

**Входные параметры (`@Input`):**
| Параметр | Тип | Описание |
|----------|-----|----------|
| `autoTests` | `'pending' \| 'passed' \| 'failed'` | Статус автотестов |
| `aiAnalysis` | `'pending' \| 'passed' \| 'failed'` | Статус AI-анализа |
| `expertReview` | `'pending' \| 'approved' \| 'rejected'` | Статус экспертной проверки |

**Структура:**
- 3 этапа с номерами (1, 2, 3)
- Соединительные линии между этапами
- Бейджи статуса под каждым этапом
- Подписи: "Автотесты", "AI-анализ", "Эксперт"

**Пример использования:**
```html
<app-review-progress 
  [autoTests]="'passed'"
  [aiAnalysis]="'passed'"
  [expertReview]="'pending'">
</app-review-progress>
```

## Компоненты страниц (Pages)

### LandingComponent
**Селектор:** `app-landing`
**Маршрут:** `/`

Главная страница платформы.

**Назначение:**
- Приветствие посетителей
- Выбор роли (кандидат или работодатель)
- Демонстрация трехэтапной системы проверки
- Преимущества платформы

**Структура:**
1. Шапка с логотипом
2. Hero-секция с заголовком
3. Две карточки выбора роли (Я ДЖУН / Я РАБОТОДАТЕЛЬ)
4. Три карточки этапов проверки
5. Три карточки преимуществ

**Переходы:**
- "Я ДЖУН" → `/junior-auth`
- "Я РАБОТОДАТЕЛЬ" → `/employer-auth`

### JuniorAuthComponent
**Селектор:** `app-junior-auth`
**Маршрут:** `/junior-auth`

Страница аутентификации для кандидатов.

**Назначение:**
- Вход существующих пользователей
- Регистрация новых кандидатов

**API Endpoints:**
- `POST /api/v1/auth/login` - вход
- `POST /api/v1/auth/register/candidate` - регистрация

**Функциональность:**
- Переключение между режимами входа/регистрации
- Валидация полей формы (email, пароль, подтверждение)
- Индикатор загрузки
- Отображение ошибок от API

**Поля формы (регистрация):**
- Имя (только при регистрации)
- Email
- Пароль (мин. 6 символов)
- Подтверждение пароля

**После успеха:** Перенаправление на `/candidate-dashboard`

### CandidateDashboardComponent
**Селектор:** `app-candidate-dashboard`
**Маршрут:** `/candidate-dashboard`

Личный кабинет кандидата.

**API Endpoints:**
- `GET /api/v1/candidates/me` - профиль
- `GET /api/v1/tasks/available` - доступные задания
- `GET /api/v1/candidates/me/tasks` - мои задания
- `GET /api/v1/candidates/me/submissions` - отправленные решения
- `PUT /api/v1/candidates/me` - обновление профиля
- `PUT /api/v1/candidates/me/skills` - обновление навыков

**Структура:**
1. **Секция профиля**
   - Аватар, имя, город
   - Описание "О себе"
   - Список навыков с уровнями
   - Кнопка "Редактировать"

2. **Боковая панель (фильтры)**
   - Фильтр по технологиям (чекбоксы)

3. **Доступные задания**
   - Список карточек заданий
   - Информация: название, компания, дедлайн, технологии
   - Ссылка на детали

4. **Задания в работе**
   - Текущие задания кандидата
   - Кнопка "Чат с компанией"

5. **Выполненные задания**
   - История отправленных решений
   - Прогресс проверки
   - Комментарии эксперта

**Модальные окна:**
- Редактирование профиля (имя, город, о себе, навыки)
- Чат с компанией

### TaskDetailComponent
**Селектор:** `app-task-detail`
**Маршрут:** `/task/:id`

Страница детальной информации о задании.

**API Endpoints:**
- `GET /api/v1/tasks/:id` - информация о задании
- `POST /api/v1/tasks/:id/take` - взять в работу
- `POST /api/v1/submissions` - отправить решение

**Query Parameters:**
- `mode` - режим отображения:
  - `available` - задание доступно (кнопка "Взять")
  - `inprogress` - в работе (форма отправки)
  - (отсутствует) - по умолчанию форма отправки

**Структура:**
1. Название задания
2. Информация: компания, дедлайн, технологии
3. Таймер обратного отсчета (дней до дедлайна)
4. Описание проекта
5. Список требований
6. Действия (зависят от mode):
   - "Взять задание" + "Задать вопрос"
   - Форма отправки решения (GitHub URL)

**Особенности:**
- Валидация URL GitHub
- Индикатор успешной отправки
- Модальное окно чата

### SubmissionResultsComponent
**Селектор:** `app-submission-results`
**Маршрут:** `/submission/:id`

Страница результатов проверки решения.

**API Endpoints:**
- `GET /api/v1/submissions/:id` - данные решения
- `GET /api/v1/submissions/:id/results` - детальные результаты

**Структура:**
1. **Заголовок**
   - Название задания
   - Дата отправки, ссылка на репозиторий
   - Прогресс проверки (3 этапа)
   - Итоговый вердикт

2. **Раздел 1: Автотесты**
   - Статистика (пройдено/всего)
   - Прогресс-бар
   - Список тестов с результатами

3. **Раздел 2: AI-анализ**
   - Количество замечаний
   - Список замечаний по категориям

4. **Раздел 3: Экспертное ревью**
   - Статус (одобрено/отклонено)
   - Комментарий эксперта
   - Или сообщение об ожидании

**Логика вердикта:**
1. Эксперт одобрил → "ОДОБРЕНО"
2. Эксперт отклонил → "ОТКЛОНЕНО"
3. AI не пройден → "ТРЕБУЕТСЯ ДОРАБОТКА"
4. Автотесты не пройдены → "НЕ ПРОШЛО АВТОТЕСТЫ"
5. Иначе → "В ПРОЦЕССЕ ПРОВЕРКИ"

## Сервисы (Services)

### ApiService
Базовый сервис для HTTP-запросов.

**Методы:**
| Метод | Описание |
|-------|----------|
| `get<T>(endpoint, params?)` | GET запрос |
| `post<T>(endpoint, body?)` | POST запрос |
| `put<T>(endpoint, body?)` | PUT запрос |
| `patch<T>(endpoint, body?)` | PATCH запрос |
| `delete<T>(endpoint)` | DELETE запрос |

**Особенности:**
- Автоматическое добавление базового URL
- JWT аутентификация через заголовки
- Таймауты запросов
- Стандартная обработка ошибок
- Поддержка query параметров

### AuthService
Управление аутентификацией.

**Методы:**
| Метод | Endpoint | Описание |
|-------|----------|----------|
| `login(credentials)` | POST /auth/login | Вход пользователя |
| `registerCandidate(data)` | POST /auth/register/candidate | Регистрация |
| `logout()` | - | Выход (очистка токена) |
| `refreshToken()` | POST /auth/refresh | Обновление токена |
| `getCurrentUser()` | GET /auth/me | Текущий пользователь |

### TaskService
Управление заданиями.

**Методы:**
| Метод | Описание |
|-------|----------|
| `getTasks(params?)` | Список заданий с фильтрацией |
| `getTaskById(id)` | Получение задания |
| `createTask(data)` | Создание задания |
| `updateTask(id, data)` | Обновление задания |
| `deleteTask(id)` | Удаление задания |
| `takeTask(id)` | Взять в работу |
| `getMyTasks()` | Мои задания (кандидат) |
| `getEmployerTasks()` | Задания работодателя |

### SubmissionService
Управление решениями.

**Методы:**
| Метод | Описание |
|-------|----------|
| `getSubmissions(params?)` | Список решений |

### CandidateService
Управление кандидатами.

**Методы:**

## Взаимодействие с API

### Базовый URL
- **Разработка:** `http://localhost:8000/api/v1`
- **Продакшен:** Относительные пути (проксирование)

### Аутентификация
Все запросы (кроме логина) требуют JWT токен:
```
Authorization: Bearer <token>
```

### Обработка ошибок
Компоненты обрабатывают следующие ошибки:
- **401 Unauthorized** - перенаправление на страницу входа
- **403 Forbidden** - сообщение "Доступ запрещен"
- **404 Not Found** - сообщение "Ресурс не найден"
- **422 Validation Error** - отображение ошибок валидации
- **500 Server Error** - сообщение "Ошибка сервера"

### Формат ответа API
```typescript
interface ApiResponse<T> {
  success: boolean;
  data?: T;
  message?: string;
  error_code?: string;
}
```

### Пагинация
```typescript
interface PaginatedResponse<T> {
  items: T[];
  total: number;
  page: number;
  size: number;
  pages: number;
}
```
